Functionality

Usability

Reliability

Performance

Supportable software can be easily updated, modified, and integrated with other systems.

Threat modeling is a systematic and proactive technique designed to identify, evaluate, and mitigate 
potential security challenges and concerns within a software system.
It involves five key steps:
Step 1: Identify Elements (Decomposition of Components and Modules)
Applications are comprised of individual features, each susceptible to attacks. 
Understanding the components of the application is crucial.

Step 2: Identify the Threats
Threats are pinpointed through a systematic review of assets, often utilizing 
models like STRIDE (details can be found below). Examples include SQL injections, 
broken authentication, and session management vulnerabilities. Risk-prone areas like poor input validations, 
over-privileged accounts, weak password policies, custom encryption, inadequate auditing, 
or error message displays should be identified.

Step 3: Document the Threats
Documentation is integral to the iterative threat modeling process. 
Architects use documentation to create a secure design/architecture and address architecture-related security threats.

Step 4: Rate Threats
Threats are rated based on risk levels, often using a High, Medium, or Low scale. 
High-risk threats require immediate attention, while 
Medium-risk threats need to be addressed with less urgency. 
Low-risk threats may be ignored based on cost and effort considerations.

Step 5: Determine Countermeasures and Mitigation
Identify countermeasures to mitigate vulnerabilities, considering a threat may exploit multiple vulnerabilities. 
This results in a list of countermeasures protecting vulnerabilities and a map showing the relationships between countermeasures and vulnerabilities. ^I2kYFH6A

Design Concepts Categories
 ^8JM9ScBJ

Design Principles are guidelines that ensure effective design and implementation of a software or system solution. 
These principles are based on industry best practices and can guide decisions throughout the development process.

Deployment Patterns are models that show how to physically structure the system for deployment. Some deployment 
patterns are useful for establishing an initial physical structure of the system in terms of tiers, while more specialized 
deployment patterns are used to satisfy quality attributes such as availability, performance, and security.

Reference Architectures

Reference Architectures are blueprints that provide an overall logical structure for specific types of applications. 
They are a reference model mapped onto one or more architectural patterns and come with a set of supporting 
artifacts that make them easy to use. Reference architectures have been proven in business and technical contexts.

Externally Developed Components refer to technologies that are not created as part of the development project. 
They can be classified into technology families, products, application frameworks, and platforms. 
Technology families represent specific technologies with common functional purposes. 
Products are self-contained functional software pieces that need minimal configuration or coding. 
Application frameworks are reusable software elements that provide generic functionality for recurring 
domain and quality attribute concerns. Platforms provide a complete infrastructure to build and execute applications.

Architectural Design Patterns are conceptual solutions to recurring design problems that exist in a defined context. 
They include patterns that address decisions at varying levels of granularity and specific patterns to address quality attributes such as security or integration.

Tactics are collections of fundamental design techniques used to achieve a response for particular quality attributes. 
They are techniques that architects have been using for years, encompassing what architects have done in practice, 
over the decades, to manage quality attribute response goals. ^LzsBbAGT

Design Principles
The principle of least privilege states that every entity should be granted only 
the minimum system resources and authorizations necessary to carry out its function within a security architecture.

Low Coupling & High Cohesion
The principle refers to maintaining a balance between dependency and autonomy among the different 
modules of a software architecture. High cohesion is achieved when a component, such as a class, contains 
methods that are all related to a single underlying concept, making it more understandable. Loose coupling 
allows components to be independent and not tied to others, resulting in increased testability and flexibility. ^swG9Osgf

SOLID Principles
 ^ptnCg9Cz

Single-Responsibility Principle (SCP)

A class or an object should have only one reason to change. This simplifies modifications and makes the code less complex. 
The cohesion is stronger, and the dependency coupling is looser.

Open-Closed Principle (OCP)

Entities should be open for extension but closed for modification. This means that they can be extended by adding what is needed, 
but they can never be modified. This significantly reduces the risk of breaking existing functionality and provides a looser coupling.

Liskov Substitution Principle (LSP)

Subtypes must be substitutable for their base types. This principle confirms that abstractions are correct and that it is 
possible to use the base type and get a correct result as the outcome.

Interface Segregation Principle (ISP)

Classes that implement interfaces should not be forced to implement methods they do not use. 
Big interfaces should be split into smaller ones so that there are no unused methods. 
This provides decoupling and easier modifications as classes only know about methods related to them.

Dependency Inversion Principle (DIP)

High-level modules should not depend on low-level modules. Both should depend on abstractions. 
Abstractions should not depend on details, but details should depend on abstractions. 
This reduces dependencies in the code modules, making it much easier to maintain if abstractions 
and details are isolated from each other. ^pvWyoUsN

Reference Architectures are methodologies that include best practices, templates, and standards based on successful past solutions. ^gT7HWp4I

Availability Tactics: Detect Faults
Fault Detection is the process of identifying faults in a system's functional units that lead to undesired or intolerable behavior 
in the entire system. However, detecting software faults can be challenging due to the complexity of large software systems.

Ping/Echo
Asynchronous request/response message pair exchanged between nodes, used to determine reachability and the round-trip delay through the associated network path.

Monitor
A component used to monitor the state of health of other parts of the system. A system monitor can detect failure or congestion in the network or other shared resources, such as from a denial-of-service attack.

Heartbeat
A periodic message exchange between a system monitor and a process being monitored.

Timestamp
Used to detect incorrect sequences of events, primarily in distributed message-passing systems.

Condition Monitoring
Checking conditions in a process or device, or validating assumptions made during the design.

Sanity Checking
Checks the validity or reasonableness of a component’s operations or outputs; typically based on knowledge of the internal design, the state of the system, or the nature of the information under scrutiny.

Voting
To check that replicated components are producing the same results. Comes in various flavors: replication, functional redundancy, analytic redundancy.

Exception Detection
Detection of a system condition that alters the normal flow of execution. For instance, system exception, parameter fence, parameter typing, timeout.

Self-Test
Procedure for a component to test itself for correct operation.

Transactions

Predictive Model
