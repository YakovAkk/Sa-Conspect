# Semi-Formal Notations: BPMN Data

> [!note] Overview
> ==Data== is one of the five BPMN symbol categories. It represents the **information needed to carry out tasks** and the **results generated by them**. Data elements are flexible — they can show data consumed by the process (input), data produced by it (output), data gathered during it (collection), or data that persists beyond the process (storage). In BPMN, ==data objects== clarify what information activities depend on, how results flow between steps, and where information is retained — making process models more accurate and self-explanatory.

---

## 1. Data Input

A ==Data Input== represents the **prerequisites of a business process** — specific data that must be available before a task or process can proceed.

### Key Characteristics

> [!info] Defining Traits
> - Visually: a **document icon with a hollow (unfilled) arrow** pointing inward — indicating that data enters the process
> - Tasks in BPMN often have **data dependencies** — they can only progress when the required input is at hand
> - Data Inputs are typically defined at the process level (not inside individual tasks)
> - Connected to activities via **Associations** (dotted lines), not Sequence Flows
> - Represents an external or pre-existing data requirement that the process consumes

> [!example] Real-World Example
> In a survey collection process, "Customer Survey Form" is a Data Input — the survey task cannot begin until the form is available. The task depends on this data being provided before execution starts.

---

## 2. Data Output

A ==Data Output== represents the **results or outcomes produced by a process**. When a task or process generates information, that information is captured as a Data Output.

### Key Characteristics

> [!info] Defining Traits
> - Visually: a **document icon with a filled (solid) arrow** pointing outward — indicating that data exits the process
> - Signals that a process has produced a result that can be consumed by downstream activities or external parties
> - Defined at the process level — typically marks the final outcome of the process
> - Connected to activities via **Associations**
> - Represents data created, transformed, or finalized by the process execution

> [!example] Real-World Example
> In a survey collection process, the "Survey Results Report" is a Data Output — it is the document produced once all survey responses are gathered and analyzed. The output is the deliverable of the process.

---

## 3. Data Object (Data Collection)

A ==Data Object== represents **information used or produced within the process flow** — at the individual task level rather than the process boundary level. It distinguishes between one-time data and ongoing data collection.

### Key Characteristics

> [!info] Defining Traits
> - Visually: a **document icon** (plain, without a bold arrow) — resembles a page with a folded corner
> - Appears inside the process flow, connected to specific tasks via Associations
> - Can represent a **single data item** (one document, one record) or a **collection** (multiple items — shown with a stacked pages icon)
> - Data Object Collection: a stack of document icons — indicates that multiple instances of data are gathered over time
> - Helps show that a task produces or consumes a specific piece or set of information

### Single vs. Collection

| Type | Visual | When to Use |
|---|---|---|
| **Data Object** | Single page icon | A single data item consumed or produced at one point |
| **Data Object Collection** | Stacked pages icon | Ongoing or batch data gathering across multiple instances |

> [!example] Real-World Example
> In a survey collection process:
> - A **Data Object** (single page) shows one completed survey form per respondent
> - A **Data Object Collection** (stacked pages) shows all survey forms gathered across all respondents over the collection period — the full dataset before aggregation

---

## 4. Data Store

A ==Data Store== represents **persistent data storage** — a place where data can be stored or retrieved, and that survives beyond the execution of a single process instance.

### Key Characteristics

> [!info] Defining Traits
> - Visually: a **database cylinder icon** (a horizontal cylinder with a double-line top — resembling a database)
> - Data stored in a Data Store **persists between process instances** — unlike data objects which exist only during a process run
> - Can be read from (a task retrieves data) or written to (a task stores data)
> - Connected to activities via **Associations**
> - Represents databases, file systems, data warehouses, CRM systems, or any persistent storage mechanism
> - Strategic decisions about what to store (and how) are part of the process design

> [!example] Real-World Example
> In an order fulfillment process:
> - A task "Look Up Customer" reads from the **Customer Database** (Data Store)
> - A task "Record Order" writes to the **Order Database** (Data Store)
>
> Both databases persist beyond any single order — they are shared across many process instances and retained long-term.

> [!tip] Storage Strategy
> Data Stores make process diagrams accurate by showing that some data is not created fresh — it is retrieved from an existing system. This is essential for modeling real integrations with databases, CRMs, ERPs, and other systems of record.

---

## 5. Data Elements — Comparison at a Glance

| Element | Visual | Scope | Direction | Persistence |
|---|---|---|---|---|
| **Data Input** | Document + hollow inward arrow | Process boundary | Into the process | Pre-exists the process |
| **Data Output** | Document + filled outward arrow | Process boundary | Out of the process | Created by the process |
| **Data Object** | Plain document page | Within process flow (task level) | In or out of tasks | Exists only during process |
| **Data Object Collection** | Stacked pages | Within process flow | In or out of tasks | Exists only during process |
| **Data Store** | Database cylinder | Cross-process | Read or write | Persists beyond the process |

---

## 6. How Data Elements Connect to the Process

> [!info] Association — the Data Connector
> Data elements are **never connected via Sequence Flow or Message Flow**. They are always connected to activities (or other flow elements) via **Associations** — dotted lines with or without arrowheads.
> - An arrowhead pointing toward the data element means the task **writes** to it (produces data)
> - An arrowhead pointing toward the task means the task **reads** from it (consumes data)
> - This keeps the data flow visually distinct from the process control flow

---

## 7. Why Data Modeling Matters in BPMN

> [!tip] Process Completeness
> Including data elements in a BPMN diagram transforms it from a flow diagram into a **complete process model**:
> - It shows **why tasks exist** — what data they need and what they produce
> - It reveals **task dependencies** — a task can't start without its required inputs
> - It documents **system integrations** — data stores show which external systems are involved
> - It clarifies **information handoffs** — data outputs show what one task produces for the next
>
> Without data elements, a diagram shows *what* happens but not *with what* — leaving implementers to guess at data dependencies.

---

## Summary

==BPMN Data elements== — Data Input, Data Output, Data Object, Data Object Collection, and Data Store — represent the information dimension of business processes. Data Inputs and Outputs operate at the process boundary; Data Objects operate at the task level within the flow; Data Stores represent persistent, cross-process storage. All are connected via Associations (dotted lines), keeping data flow visually distinct from control flow. Together they make BPMN diagrams complete — showing not just the sequence of activities, but the information those activities depend on, produce, and store.
