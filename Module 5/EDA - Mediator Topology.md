# Event-Driven Architecture: Mediator Topology

> [!note] Overview
> The ==mediator topology== is used when an event requires **multiple steps that must be organized in a specific order**. A central ==mediator== orchestrates the flow, deciding the sequence of steps and which can run in parallel.
>
> **Example:** Placing a stock trade involves checking the trade, following compliance rules, choosing a broker, calculating fees, and executing the trade — all in a controlled order.

## The Four Main Components

### 1. Events

> [!info] Two Types of Events
> - **Initial event** — the original event the mediator receives from an external source (e.g., "place stock trade").
> - **Processing events** — generated by the mediator and transmitted to individual event-processing components for each step in the workflow.

The mediator translates one initial event into many smaller processing events, each representing a single step.

### 2. Queues

> [!info] Event Queues
> Having anywhere from **a dozen to several hundred event queues** is commonplace in EDA. The event queue component is flexible and can be implemented as:
> - A **message queue** (point-to-point delivery)
> - A **web service endpoint**
> - A **combination of both**
>
> Queues pass messages between components, decoupling producers from consumers.

### 3. Mediator

> [!info] The Event Mediator
> The ==event mediator== is the **orchestration core** of this topology:
> - Receives the **initial event**.
> - Knows all the steps required to process it.
> - For each step, dispatches a specific **processing event** to the appropriate ==event channel==.
> - Controls **order of execution** and determines which steps can run **simultaneously** (in parallel).

The mediator does not execute business logic itself — it only coordinates the flow.

### 4. Event Channels

> [!info] Event Channels
> ==Event channels== are the means by which the mediator **asynchronously transmits** processing events to event processors.
> - Can be **message queues** (point-to-point) or **message topics** (publish-subscribe).
> - ==Message topics== are the more common choice in the mediator topology, allowing multiple processors to subscribe to the same step if needed.
> - Channels decouple the mediator from the processors — the mediator does not wait for a response before moving to the next step.

### 5. Event Processors

> [!info] Event Processors
> ==Event processors== encapsulate the **application business logic** for handling a specific processing event.
>
> Key characteristics:
> - **Self-contained** — each processor owns everything needed to execute its task.
> - **Independent** — processors do not call or depend on other processors.
> - **Highly decoupled** — they only know about their input event, not the broader workflow.
> - **Single responsibility** — each processor executes **one business task**.

## Component Summary

| Component | Role |
|---|---|
| **Initial Event** | Triggers the entire workflow |
| **Processing Events** | Represent individual steps generated by the mediator |
| **Event Queues** | Transport messages between components; flexible in implementation |
| **Mediator** | Orchestrates step order; dispatches processing events |
| **Event Channels** | Async conduit between mediator and processors (topics or queues) |
| **Event Processors** | Execute business logic for a single step; self-contained |

## How the Flow Works

```
External Trigger
      ↓
 Initial Event
      ↓
  [Mediator]  ← knows the full workflow
  /    |    \
  ↓    ↓    ↓   (processing events dispatched to channels)
[Ch1][Ch2][Ch3]  ← Event Channels (async)
  ↓    ↓    ↓
[EP1][EP2][EP3]  ← Event Processors (business logic)
```

> [!tip] Parallel Execution
> The mediator can dispatch multiple processing events simultaneously when steps are independent. This reduces total processing time compared to a fully sequential approach.

## Strengths and Trade-offs

| Strengths | Trade-offs |
|---|---|
| Clear, controlled workflow — easy to trace | Mediator is a potential bottleneck |
| Supports parallel step execution | Mediator becomes a single point of failure |
| Business logic isolated in each processor | Mediator logic can grow complex over time |
| Flexible queue implementation | Harder to make the mediator itself highly available |

## When to Use Mediator Topology

> [!example] Good Fit
> - Complex workflows with **multiple ordered steps** (e.g., order processing, financial transactions, multi-stage approval flows).
> - When some steps must run **sequentially** and others can run **in parallel**.
> - When **central oversight and traceability** of the full workflow are important.
> - When you need to **easily modify the workflow** without changing individual processors.

## Summary

The mediator topology provides a structured, orchestration-based approach to event-driven processing. By centralizing workflow control in the mediator while keeping business logic isolated in independent event processors, it handles complex multi-step processes cleanly. The trade-off is the mediator's centrality — it must be highly available and carefully designed to avoid becoming a bottleneck as the workflow grows.
